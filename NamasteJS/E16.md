### JavaScript Runtime Environment

JRE: JS Engine + Web APIs + Callback Queues + Microtask Queues + Event Loop and more interesting things.

There are various JREs. Most common is the Browser. 

Node.js is the Runtime Environment that has everything required as a runtime and can be ran outside the browser as well.

### Various JS Engines:

First JS engine developed was SpiderMonkey. SpiderMonkey is the engine used by Mozilla Firefox.

Fastest JS Engine currently is V8. V8 is used by chrome, Node.js and Deno as well.

Other engines are Chakra, Tamarin etc.

### JS Engine:

JS Engine is the heart of JRE.

It executes JS code.

It majorly contains two things: Call Stack and Memory Heap.

It executes the JS code in 3 steps.

1. Parsing: First the JS engine converts the JS code into tokens. Then it converts it into "Abstract Syntax Tree" known as AST. It is basically a syntax parser. Refer [this](https://astexplorer.net/)

2. Compilation: JavaScript has JIT Compilation i.e. Just In Time Compilation. It uses both Interpreter and Compiler for compilation of the code. All JS Engines have own algorithms of compilations. All these JS engine's job is to optimize the code as much as it can. 

3. Execution: Memory Heap is space where functions and vars are assigned memory and works in sync with the Call stack. Mark and Sweep Algorithm is famous garbage collector algorithm. Inlining, Copy Elision, Inline Caching etc. are imp compiler concepts.

### V8 Engine

V8 Interpreter: Ignition -> Byte Code
V8 compiler: TurboFan -> Creates Optimized Machine Code -> Byte code

Compiler Optimizes and Interpreter executes.

Garbage collector: Orinoco, Oilpan

# JavaScript Execution Pipeline Notes

JavaScript code does **not execute directly**.
It goes through multiple stages inside the JavaScript engine (like **V8 JavaScript Engine**).

Main stages:

```
JavaScript Code
      ↓
Parsing
      ↓
Compilation (Creation Phase)
      ↓
Execution Phase
      ↓
JIT Optimization (only for hot code)
```

---

# 1️⃣ Parsing

Parsing is the **first step** where the engine reads the JavaScript code and understands its structure.

### Steps in Parsing

## 1. Lexical Analysis (Tokenization)

The code is broken into **tokens**.

Example:

```javascript
let sum = 10 + 20;
```

Tokens produced:

```
let
sum
=
10
+
20
;
```

Each token represents something meaningful:

| Token Type | Example |
| ---------- | ------- |
| Keyword    | `let`   |
| Identifier | `sum`   |
| Operator   | `+`     |
| Literal    | `10`    |

---

## 2. Syntax Analysis

The parser checks if the tokens follow **JavaScript grammar rules**.

Then it creates an **AST (Abstract Syntax Tree)**.

Example AST (simplified):

```
AssignmentExpression
   |
   +-- Identifier(sum)
   |
   +-- BinaryExpression(+)
        |
        +-- 10
        +-- 20
```

If syntax is invalid:

```javascript
let = 10;
```

Error:

```
SyntaxError
```

Execution **never starts** if parsing fails.

---

# 2️⃣ Compilation (Creation Phase)

After parsing, JavaScript performs **initial compilation**.

This phase:

* creates **scope**
* allocates **memory**
* performs **hoisting**
* registers variables and functions

This step **always happens**, even for small programs.

---

## Memory Creation

Example:

```javascript
var a = 10;
function test(){}
```

During compilation:

```
Global Memory
a → undefined
test → function definition
```

---

## Hoisting

Hoisting occurs because the engine **scans the entire code during compilation**.

Variables and functions appear to move to the top.

Example:

```javascript
console.log(a);
var a = 10;
```

Internal view:

```javascript
var a;
console.log(a);
a = 10;
```

Memory during creation phase:

```
a → undefined
```

Execution output:

```
undefined
```

---

## Function Hoisting

Functions are **fully hoisted with their definition**.

Example:

```javascript
sayHi();

function sayHi(){
  console.log("hi");
}
```

Memory:

```
sayHi → function definition
```

Output:

```
hi
```

---

## Function vs Var Priority

Function declarations have **higher priority than `var`** during hoisting.

Example:

```javascript
console.log(a);
var a = 10;
function a(){}
```

Compilation phase:

```
a → function a(){}
```

Execution:

```
console.log(a) → function a(){}
a = 10
```

Output:

```
function a(){}
```

---

## let and const Hoisting (Temporal Dead Zone)

`let` and `const` are also hoisted but **not initialized immediately**.

Example:

```javascript
console.log(a);
let a = 10;
```

Error:

```
ReferenceError: Cannot access 'a' before initialization
```

This time period is called:

```
Temporal Dead Zone (TDZ)
```

---

# 3️⃣ Execution Phase

After compilation, the engine runs the code **line by line**.

Example:

```javascript
var a = 10;
var b = 20;
console.log(a + b);
```

Execution:

Step 1

```
a = 10
```

Step 2

```
b = 20
```

Step 3

```
console.log(30)
```

Output:

```
30
```

---

# 4️⃣ Interpreter vs Compiler in JavaScript

JavaScript engines use **both interpreter and compiler**.

---

## Interpreter

An interpreter executes code **line by line immediately**.

Example flow:

```
Read line → Execute → Next line
```

Advantages:

* Fast startup
* No need to compile entire program

Example interpreter in **V8 JavaScript Engine**:

```
Ignition
```

---

## Compiler

A compiler converts code into **machine code before execution**.

Traditional compiled languages:

* **C++**
* **Java**

Advantages:

* Faster execution after compilation

---

# 5️⃣ JIT (Just-In-Time Compilation)

Modern JS engines combine **interpreter + compiler**.

This technique is called:

```
JIT Compilation
```

Used by **V8 JavaScript Engine**.

Flow:

```
JS Code
  ↓
Parser
  ↓
Initial Compilation
(scope + hoisting)
  ↓
Interpreter executes code
  ↓
Engine monitors performance
  ↓
Hot code detected
  ↓
JIT Optimizing Compiler
```

---

# 6️⃣ Hot Code Concept

The engine tracks functions.

If a function runs **many times**, it becomes:

```
Hot Code
```

Example:

```javascript
function add(a,b){
  return a+b;
}

for(let i=0;i<1000000;i++){
  add(1,2);
}
```

Engine detects:

```
add() executed many times
```

Then it sends the function to an **optimizing compiler**.

In **V8 JavaScript Engine**:

| Component | Purpose             |
| --------- | ------------------- |
| Ignition  | Interpreter         |
| TurboFan  | Optimizing Compiler |

---

# 7️⃣ Small Programs and JIT

Small programs usually run only once.

Example:

```javascript
console.log(10 + 20);
```

Execution:

```
Parser
↓
Interpreter
↓
Done
```

JIT optimization **does not happen** because the code is not executed repeatedly.

However:

✔ **Initial compilation still happens**
✔ **Hoisting still works**

---

# 8️⃣ Two Types of Compilation in JavaScript

This is the **key concept**.

| Type                | Description                         | Happens Always?                   |
| ------------------- | ----------------------------------- | --------------------------------- |
| Initial Compilation | Parsing + scope creation + hoisting | Yes                               |
| JIT Optimization    | Recompiling hot code for speed      | Only for frequently executed code |

---

# 9️⃣ Complete JavaScript Engine Flow

```
JavaScript Code
        ↓
Parsing
(tokenization + syntax analysis)
        ↓
AST (Abstract Syntax Tree)
        ↓
Initial Compilation
(scope + memory creation + hoisting)
        ↓
Execution (Interpreter)
        ↓
Hot code detection
        ↓
JIT Optimizing Compilation
```

---

# 🔑 Key Interview Takeaways

1. JavaScript is **not purely interpreted**.
2. It uses **interpreter + compiler (JIT)**.
3. **Hoisting happens during the compilation/creation phase.**
4. **Initial compilation always happens**, even for small programs.
5. **JIT optimization happens only for frequently executed (hot) code.**
